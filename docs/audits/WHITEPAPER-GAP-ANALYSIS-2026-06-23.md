# Vosj Community Edition — Whitepaper Gap Analysis

**Date:** 2026-06-23 · **Method:** a 10-area multi-agent analysis (workflow `wf_041273b7`) that extracted the load-bearing claims of `site/whitepaper.html` (the "statement of understanding") and mapped each to the actual code — `implemented / partial / missing / divergent / aspirational` with file evidence and gap severity.

## Verdict

SUBSTANTIATED WITH ONE MATERIAL DEFECT. The whitepaper's headline claim — a fail-closed, verified-before-Jump, human-signed, tamper-evident migration factory — is genuinely realized in code as structural (not procedural) guarantees: I verified directly that the cutover gate is engine-injected and non-removable (state-machine.js INJECTED_CUTOVER_GATE), fails closed without proof.ok===true (gate.js L44-50, cutoverUnit L95), rejects every non-human and self-signing signer at the contract layer (contracts/index.js assertHumanIndependent L41-49), records every transition in an HMAC-SHA256 hash-chained ledger that throws on a missing key rather than falling back (ledger.js _key L34-40), and physically denies big-bang to Refactor/Replatform/Relocate via a frozen contract table (disposition.js L42/58/67). These are the load-bearing claims and they hold. The defect: the paper's *distinctive* claim that 7-R disposition is enforced as a gate input is NOT realized — disposition is persisted (routes.js buildWorkload L93) but no code computes phase-gate criteriaMet, so the gate signer's only criteria check (gate.js L40, rejects on criteriaMet===false) never fires for the P2 kickoff gate. A wave can be signed through P2→P3 with zero workloads dispositioned. This undermines the paper's positioning of disposition as a hard gate, though it does NOT undermine the verified-before-Jump cutover guarantee, which is independently enforced. Verdict: the safety floor is real and excellent; the disposition-gate enforcement and template lifecycle must close before the CE fully matches its own whitepaper.

## Overall maturity: **63%**

The Vosj Community Edition is a genuine, code-true implementation of the whitepaper's *governance core* — the irreversible safety machinery — but a deliberate and partial implementation of the *factory* surrounding it. The four invariants that carry the headline claim (no-agent-self-sign, separation of duties, verified-before-Jump, tamper-evident HMAC ledger) plus the 7-R Strangler-Fig structural guarantee are realized as non-bypassable structural guarantees in frozen data and fail-closed code paths — not policy, not convention, not review. However, the implementation has one true CE defect that materially weakens the paper's distinctive claim (the 7-R disposition is *stored* but never *enforced as a gate input*: phase-gate `criteriaMet` is never computed, so 'no kickoff until every workload carries a disposition' is decorative caf.json text), and the framework-template engine ships read-only against a fully-prepared but unused persistence schema (no clone/create — the paper's 'methodology becomes configuration' is half-built). The autonomous fleet, digital twins, credential brokering, hardened runtime, and self-improvement loop are honestly and explicitly scoped to Enterprise Edition in README/CORE-IDEA — these are legitimate scope boundaries, not gaps. Net: the headline claim of a fail-closed, verified-before-Jump, human-signed, tamper-evident migration factory is substantiated at the gate/ledger/cutover layer with rigor that exceeds most production systems; it is the *disposition-as-gate-input* enforcement and the *consultant template lifecycle* that fall short of the paper inside the CE's own declared scope.

## Critical gaps — true CE defects that contradict the paper

- ❌ §7 disposition-as-gate-input is NOT enforced (true CE defect undermining the paper's *distinctive* treatment): disposition is persisted in routes.js buildWorkload (L93) but no code computes phase-gate criteriaMet; gate.js only rejects on criteriaMet===false (L40), which is never set for the P2 'kickoff complete' gate. The caf.json criterion 'every in-scope workload carries a disposition' is unverified text. A wave can be transitioned P2→P3 (and onward) with zero dispositions assigned. This is the single gap that contradicts a headline-adjacent claim within CE scope.
- ❌ §13.2 revocable acceptance + three-way contingency reversal is entirely absent in CE. The whitepaper's 'Net effect: provably equivalent before traffic moves AND reversible if not' is only half-true — the proven-before half is enforced; the reversible half (30-min revocation window, post-cutover drift revoke, three-way reversal authority) has no code, no schema column, and no state. cutoverUnit is a single irreversible forward transition.
- ❌ §15.2 fail-closed vault is specification-only in CE: VAULT_MASTER_KEY is parsed in config.js but never consumed — there is no encryption/decryption code, no credentials/vault table, no KDF. The paper frames this under Invariant 5 (fail-closed), so the *absence of any vault* means the 'encrypted credential vault, fail-closed' claim is unrealized in the CE codebase even though the ledger fail-closed sibling IS real. (Mitigated: CE is BYO-AI and credentials are operator-supplied K8s secrets, but the paper's stated vault control does not exist.)

## Strengths — whitepaper claims realized as STRUCTURAL guarantees in code

- ✅ Verified-before-Jump is STRUCTURAL and non-removable: state-machine.js injects INJECTED_CUTOVER_GATE on every template (L18-26, L120) and cutoverUnit/gate.sign fail closed unless proof.ok===true AND proof.hash exists (state-machine L95, gate L44-50). A non-verified cutover is an unreachable state, exactly as claimed.
- ✅ No-agent-self-sign and separation-of-duties are enforced in the contract base class (contracts/index.js assertHumanIndependent L41-49: rejects signer.kind!=='human' and author===signer.id), invoked on every sign() and re-checked at the MCP seam — structurally impossible to bypass, not policy.
- ✅ Tamper-evident ledger is real: HMAC-SHA256 over (prevHash + canonical-JSON), hash-chained, verifyChain detects forge/back-date (ledger.js L52-94), key custodied outside the DB and FAIL-CLOSED (throws on missing key, never a dev fallback, L34-40).
- ✅ 7-R Strangler-Fig structural guarantee is genuine: Refactor/Replatform/Relocate are frozen to cutoverStyle:STRANGLER_FIG with bigBangAvailable:false derived from the immutable contract table (disposition.js L42/58/67/87-88) — big-bang is physically unselectable for high-risk dispositions, enforced by data structure not reviewer diligence.
- ✅ Hard invariants are structurally UNWAIVABLE: waiver.js HARD_INVARIANT_CHECKS + isWaivable default-deny refuse any waiver against verified-before-cutover, no-self-sign, separation-of-duties, ledger-fail-closed, and baseline-drift — defence-in-depth above the primary enforcement in gate/ledger/reconcile.
- ✅ Baseline-drift guard is fail-closed: a missing or stale baseline forces proof.ok=false (reconcile.js isFreshBaseline L31-36, L69), and it is on the non-waivable list — a stale baseline cannot be overridden into a cutover.
- ✅ Template-driven signed-gate FSM is data-driven not constant-driven: signTransition reads allowed transitions and signer roles from the bound template, RBAC capability derived from the template role, version pinned at kickoff (routes.js L123-125) so a later edit cannot mutate an in-flight run.
- ✅ Durable order queue uses correct atomic-claim semantics (FOR UPDATE SKIP LOCKED) so concurrent stations cannot double-execute — the coordination-without-shared-state claim is real.
- ✅ Defaults deny throughout: requireAuth returns 503 when no token configured, every mutation requires an explicit {domain}:{resource}:{action} capability, and agents are minted without any sign-as-role grant.

## CE vs Enterprise

LEGITIMATELY ENTERPRISE-SCOPED (honestly documented in README L40-44/L143 and CORE-IDEA L79-90/L108 — NOT gaps): (1) the managed autonomous fleet of role-specialized coding agents and their work loops; (2) per-engineer AI digital twins / off-hours operation; (3) the self-improvement 'dreaming' learning loop that feeds post-mortems back into templates and agent priors (§11.3); (4) the runtime-isolation layer — hardened RuntimeClass (gVisor/Kata/Firecracker), credential broker/sidecar, short-TTL audience-scoped token vending (§10.4, §15.6, §15.8 egress policy); (5) SSO, enterprise RBAC, hosted audit, support/SLA; (6) Gate Council operating model, RACI tracking, 8-hourly control-testing cadence (§12.4-12.5 procedural/org layer); (7) PostgreSQL quorum-sync replication / WAL / PITR (deployment-infra, not app code). The CE correctly ships the open *plumbing* (MCP server, devstation pod substrate, engine, gates, ledger, connectors, reconciliation) and the BYO-AI seam — this is a coherent, defensible boundary.

TRUE CE GAPS (inside the CE's own declared scope, must be fixed to match the whitepaper): (1) §7 disposition-as-gate-input enforcement — the engine and caf.json are CE, the missing criteriaMet computation is a CE defect, not EE; (2) §8 framework-template clone/create/edit/diff lifecycle — the schema is shipped in CE and the paper lists 'consultant experience' as a CE capability ('templates, MCP server ... all open in CE'), so read-only-only is a CE shortfall; (3) §15.2 the fail-closed encrypted vault — explicitly listed as CE-included in CORE-IDEA L79/L89 ('fail-closed vault ... all open in CE') yet has zero implementation; (4) §13.2 revocable acceptance / three-way reversal — the paper presents this as part of the core verification guarantee, not an EE add-on; (5) §12.6 evidence-package export + control-mapping manifest — CORE-IDEA L79/L89 lists 'the exportable evidence package' as CE-included. These five are the line between marketing and code that the CE must close to be self-consistent with its own positioning.

### MCP Server Control Plane, Order Dispatch, and Security (§9 + §9.2 + §9.3 + §12 Invariants + §14 Data Model + §15 Security)

**Whitepaper:** §9 (The MCP Server — Control Plane for Tools and Orders); §9.1 (Why MCP is the right substrate); §9.2 (Order dispatch — a composition, not an MCP property); §9.3 (Security model summary); §12 (Governance & Safety Invariants, Invariants 1-6); §14.1 (The signed-gate state machine); §14.2 (MCP persistence); §14.4 (Durability & key custody); §15 (Security & Compliance Architecture) · **Maturity: 62%**

The CE implements the core MCP server, durable order queue, gate-signing engine, and most load-bearing security invariants with high fidelity. The security invariants that do not depend on devstations are fully implemented; those that assume a running fleet (e.g., per-identity segmentation, brokered short-lived credentials, hardened runtime isolation) are architectural rather than operational gaps — they describe what must be added when deploying agents against the CE's control plane.
